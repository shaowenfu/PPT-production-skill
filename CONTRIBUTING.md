# Contributing

## 目的

本文件定义向本仓库提交变更时的最小约束。

这里的前提已经变了：

- 这个仓库是专门面向 `PPT production workflow`（PPT 制作工作流）的
- 不是通用 `tooling repository`（工具仓库）
- 因此所有提交都必须直接服务于这条工作流

目标只有三个：

- 保持工作流边界清晰
- 保持契约和实现一致
- 保持端到端可运行

## 提交前先判断

提交任何变更前，先问自己 4 个问题：

1. 这个改动是否直接服务于 `PPT workflow`？
2. 它是在改 `spec`（规范）、改 `program`（程序），还是两者都要改？
3. 它会不会影响 `state.json`、CLI、产物结构或状态流转？
4. 它是否会破坏端到端链路？

如果第 1 个问题答不上来，就不应该提交到这个仓库。

## 本仓库允许的变更类型

适合进入本仓库的内容：

- `docs/` 中与 PPT 工作流直接相关的规范、设计、契约
- `scripts/` 中的工作流程序
- `pptflow/` 中的共享模块
- `PPT/` 下用于示例、调试或验证的项目工作区产物
- 与上述内容直接相关的测试、模板、样例

不适合进入本仓库的内容：

- 与 PPT 无关的通用能力
- 某个独立产品的业务逻辑
- 长期运行的服务端应用
- 一次性且不可复用的临时脚本
- 没有契约、没有边界、只能靠作者自己理解的黑盒逻辑

## 变更优先级

### 1. 改行为，先改文档

如果你修改以下任意内容，必须同步更新 `docs/`：

- 工作流步骤
- 状态机
- `state.json schema`
- CLI 参数
- 输入输出文件结构
- A/B 页面策略
- 错误语义

不要让实现先漂走，再指望别人从代码里反推规范。

### 2. 改状态，必须看状态机

如果你修改：

- `current_state`
- `transition_history`
- `artifacts`
- `retry_count`
- 回退规则

那你改的不是“一个字段”，而是在改工作流控制面，必须同时检查：

- `PPT-workflow-spec`
- `state-json-schema`
- 相关程序的状态读写逻辑

### 3. 改程序，必须考虑端到端链路

本仓库里的程序不是孤立命令，而是链式工作流节点。

因此修改任一程序时，至少要检查它的：

- 上游输入是否仍兼容
- 下游产物是否仍兼容
- `stdout JSON summary` 是否仍兼容
- 退出码是否仍兼容
- 状态写回是否仍兼容

## 新增或修改脚本时必须补齐的内容

每个脚本至少要能回答这些问题：

### 1. Purpose

- 这个程序负责哪一步
- 它做什么
- 它明确不做什么

### 2. Inputs

- CLI 参数
- 输入文件
- 依赖的环境变量
- 前置状态

### 3. Outputs

- 输出文件
- `stdout JSON summary`
- 状态写回
- 退出码

### 4. Error Semantics

- 输入错误怎么报
- 环境错误怎么报
- 外部模型或服务错误怎么报
- 输出校验失败怎么报

禁止静默吞错。

### 5. Verification

至少要给出一种验证方式：

- `py_compile`
- 单脚本运行
- `openclaw_simulator.py` 端到端验证
- 其他可复现验证步骤

## 修改 docs 时的要求

`docs/` 不是背景材料，而是正式契约层。

因此修改文档时要满足：

- 和当前代码行为一致
- 和当前目录结构一致
- 和当前状态机一致
- 用语稳定，不混用旧口径和新口径

例如当前正式口径已经是：

- 所有页面都生成图片
- A/B 区别是 `prompt strategy`
- 不存在 A/B 执行分支

那文档里就不能再出现“B 类页才生成图片”这种旧描述。

## 修改 `pptflow/` 时的要求

`pptflow/` 是共享基础层，改这里要格外克制。

提交前至少确认：

- 有没有破坏已有脚本导入
- 有没有破坏共享 `schema`
- 有没有破坏错误映射
- 有没有破坏 `OpenAI SDK` 封装
- 有没有破坏 `ppt_builder` 的函数契约

如果一个改动会波及多个脚本，优先先收敛接口，再分步修改调用方。

## 修改 `scripts/` 时的要求

每个程序都应该尽量保持：

- 单一职责
- 可单独运行
- 可被 `openClaw` 调用
- 可通过 `stdout JSON summary` 判断结果

尽量不要把复杂共享逻辑直接堆进脚本里；共享逻辑应下沉到 `pptflow/`。

## 推荐自检清单

提交前至少做以下自检：

1. `source venv/bin/activate`
2. `python -m py_compile scripts/*.py pptflow/*.py`
3. 跑受影响的单脚本
4. 如果改动影响主链路，跑一次 `openclaw_simulator.py`
5. 检查 `docs/` 是否同步更新

如果改动涉及：

- 页面规划
- Prompt 生成
- 图片生成
- PPT 装配
- 状态流转

默认应做一次端到端验证。

## 最小提交标准

一个合格提交至少应满足：

- 改动明确服务于 PPT 工作流
- 文档和实现一致
- 输入输出契约没有悬空
- 错误语义仍可分支处理
- 关键链路可验证

如果这些条件不满足，就说明这次提交还没有收敛好，不应进入主干。
