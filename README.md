# Claw-assistant

## 仓库定位

`Claw-assistant` 是一个 `tooling repository`（工具仓库），不是业务产品仓库。

它的职责是沉淀两类资产：

- `Skill docs`（技能文档 / SOP 员工手册）：定义某类任务该怎么做、输入输出是什么、边界在哪里、执行时要遵守什么约束。
- `Python scripts`（Python 脚本 / 机械臂）：负责执行重复、具体、偏脏活累活的操作，把可编排的动作真正落地。

在开发这些能力资产时，仓库可以临时使用 `Tasks/` 作为工作目录，存放需求文档、设计文档、参考资料、草稿和中间产物。但这只是开发过程的脚手架，不属于仓库正式能力体系的一部分；当能力稳定后，这部分内容应被提炼或移除。

在整个体系里：

- `openClaw` 是“大脑 + 大堂经理”：负责理解目标、拆解任务、选择合适的能力、发起调度。
- `Skill docs` 是 `SOP`（标准作业流程）：负责把经验固化成可复用、可传达、可约束的规则。
- `Python scripts` 是“机械臂”：负责执行，少决策，多落地。
- 这个仓库负责存放和维护这些能力资产本身。

一句话概括：这个仓库不是用来“做产品”的，而是用来“制造和维护可被调度的工具能力”的。

## 设计原则

### 1. 先有定位，再有实现

新增内容前，先回答两个问题：

1. 这是在增加一个可复用的能力，还是在写某个具体业务？
2. 这项能力应该沉淀为 `Skill doc`，还是落地为 `Python script`？

如果它不能被复用、不能被调度、不能被说明清楚，就不应该进入这个仓库。

### 2. 文档与执行分离

`Skill docs` 负责定义“应该做什么”和“如何判断做得对”。

`Python scripts` 负责定义“具体怎么执行”。

不要把大段执行逻辑塞进文档，也不要把流程约束、输入输出契约只藏在脚本实现里。文档和脚本必须各司其职，但相互对应。

`Tasks/` 中的工作文件可以作为设计输入，但不能替代正式文档和实现契约。临时草稿必须在沉淀为稳定能力前，被提炼进 `Skill docs`、脚本说明或仓库规范中；能力稳定后，对应 `Tasks/` 内容应可删除。

### 3. openClaw 负责编排，不负责干脏活

`openClaw` 应该尽量保持在 orchestration（编排）层：

- 理解用户目标
- 选择合适的 `Skill`
- 调用合适的脚本
- 汇总结果并继续推进任务

不应该把本应沉淀为脚本的重复动作重新塞回 prompt（提示词）或临时操作里。

### 4. 先定义契约，再写脚本

每个能力在实现前，至少应明确：

- 它解决什么问题
- 输入是什么
- 输出是什么
- 失败时如何表达错误
- 哪些属于非目标（non-goals，非目标）

脚本是对契约的实现，不是契约本身。

### 5. 保持工具原子化

一个脚本只解决一个清晰的问题，一个 `Skill` 只描述一类清晰的任务。

避免出现：

- 一个脚本承担多个互不相关职责
- 一个文档同时覆盖多个边界不同的工作流
- 用过多参数去兼容并不存在的未来需求

能拆成两个简单能力时，不要硬做成一个复杂能力。

### 6. 优先局部性与可理解性

理解一个能力所需的上下文应尽量集中：

- 文档中能看出用途、边界、输入输出
- 脚本中能看出执行路径和错误语义
- 文档与脚本的映射关系应清晰可追踪

减少“文档在这里、实现散落 elsewhere（别处）、约束靠猜”的情况。

### 7. 错误必须可枚举、可定位

工具仓库不是“跑通一次就行”，而是“长期可调用”。

因此脚本必须尽量提供稳定的错误语义：

- 失败要显式
- 错误信息要带必要上下文
- 不静默吞错
- 不把内部实现细节原样暴露成外部契约

如果某个脚本失败了，调用方应该能知道是输入问题、环境问题、依赖问题，还是目标对象本身的问题。

### 8. 让能力可以自证正确

一个可复用工具不能依赖作者本人记忆来证明自己可用。

因此关键能力应逐步补齐：

- 最小可运行示例
- 基本测试或断言
- 必要的日志与输出约定
- 清晰的使用说明

目标不是“完美工程化”，而是“脱离作者本人后仍可验证”。

## 什么应该放进这个仓库

适合长期保留在本仓库正式能力体系中的内容：

- 可被 `openClaw` 或其他调度方复用的 `Skill docs`
- 可独立执行、职责清晰的 `Python scripts`
- 工具使用说明、输入输出契约、约束说明
- 与这些能力直接相关的测试、样例、模板

开发期可以临时放在仓库中的内容：

- 放在 `Tasks/` 下的需求文档、设计文档、参考资料、草稿和中间产物
- 用于孵化新能力的工作材料

不适合进入本仓库的内容：

- 某个具体产品的业务代码
- 与工具能力无关的长期运行服务
- 只服务单次任务、没有复用价值的临时脚本
- 不能说明输入输出和边界的“黑盒经验”

## 推荐目录思路

当前仓库还很轻，目录可以随着能力增长逐步演化，但建议遵循下面的方向：

```text
.
├── README.md
├── CONTRIBUTING.md
├── Tasks/             # working area / 开发期脚手架，可移除
├── skills/            # Skill docs / SOP
├── scripts/           # Python scripts / 机械臂
├── templates/         # 模板、样例输入输出
├── tests/             # 对关键脚本的验证
└── docs/              # 补充设计说明
```

目录设计原则只有一个：稳定能力按“能力资产类型”组织；`Tasks/` 只是开发期工作区，不属于正式能力分层，能力稳定后应清理或移除。

## Tasks 的定位与边界

`Tasks/` 的作用是为新能力开发提供脚手架，典型内容包括：

- 原始需求文档
- 设计文档
- 参考资料
- 中间产物
- 头脑风暴记录与草稿

`Tasks/` 的特点是：它服务于能力开发过程，而不是能力体系本身。也就是说，它可以存在，也可以在能力稳定后被整体删除，不应成为调用方理解能力的必要前提。

但 `Tasks/` 不是：

- 正式需求契约
- `Skill` 的定义位置
- 脚本输入输出的权威说明
- 仓库正式 `tooling repository` 分层的一部分

也就是说，工具开发可以从 `Tasks/` 出发，但不能把能力长期绑定在 `Tasks/` 上。一旦某个想法要变成长期可复用能力，就必须把稳定部分提炼到正式文档和实现中，并让 `Tasks/` 中对应材料变成可删除的开发脚手架。

## 新增能力时的判断准则

当你准备往仓库里加东西时，建议先检查：

1. 这个东西是否会被复用至少第二次？
2. 它当前是否还只是 `Tasks/` 中的开发材料，还是已经值得沉淀成稳定能力？
3. 它的输入输出是否可以说清楚？
4. 它应该是 `Skill doc`、`Python script`，还是二者都需要？
5. 如果未来完全重写实现，外部调用方式是否可以保持不变？
6. 它是否引入了不必要的参数、分支和未来假设？

只要这些问题里有多个答不上来，就说明这个能力还不适合沉淀进仓库。

## 非目标

这个仓库当前的非目标非常明确：

- 不做业务产品主仓库
- 不承载复杂、长生命周期的应用服务
- 不为了“以后也许会用”而提前设计重型框架
- 不把 `Tasks/` 中的开发脚手架直接当成正式契约
- 不把调度逻辑、说明文档、执行脚本搅成一团

## 核心判断标准

衡量这个仓库是否健康，不看脚本数量，也不看文档数量，而看三件事：

- `openClaw` 是否更容易调用这些能力
- 新能力是否更容易被理解、复用、替换
- 失败时是否更容易定位和修复

如果答案是肯定的，这个仓库就在朝正确方向演化。
